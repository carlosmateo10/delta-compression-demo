Delta Compression for Dynamic HTML

Using previous responses as compression dictionaries for content served with Cache-Control: no-cache

The gap in RFC 9842

RFC 9842 (Compression Dictionary Transport) enables two use cases: delta compression for versioned static resources (using app.v1.js as a dictionary for app.v2.js) and pre-built shared dictionaries for dynamic content (a separately authored dictionary file fetched during idle time).

Neither approach addresses the largest class of dynamic web content: pages served with max-age=0 or no-cache — search results, news feeds, social media timelines, dashboards. Under RFC 9842, a response's usability as a dictionary is tied to its HTTP cache lifetime. When a page expires immediately, the browser discards it from the dictionary cache before the user navigates to the next page. There is no dictionary to reference.

What Dictionary TTL changes

Dictionary TTL is an experimental extension to the Use-As-Dictionary header that decouples dictionary lifetime from HTTP cache lifetime: 

Use-As-Dictionary: match="/search*", ttl=600

This tells the browser: keep this response usable as a dictionary for 600 seconds, regardless of Cache-Control. A dynamic page served with max-age=0 can now survive long enough to serve as a dictionary for the next navigation.

Protocol flow

Browser                                Server
  │                                        │
  │  GET /search?q=cats                    │
  │───────────────────────────────────────>│
  │                                        │  generate, compress, cache
  │  Content-Encoding: br                  │
  │  Use-As-Dictionary: match="/*",ttl=600 │
  │  Content-Dictionary: :<b64 SHA-256>:   │
  │<───────────────────────────────────────│
  │                                        │
  │  [browser caches as dictionary]        │
  │                                        │
  │  GET /search?q=dogs                    │
  │  Available-Dictionary: :<b64 SHA-256>: │
  │───────────────────────────────────────>│
  │                                        │  look up dict, compress delta
  │  Content-Encoding: dcb                 │
  │<───────────────────────────────────────│

How to use this demo

Search for anything on the main page. The first page shows standard and static shared dictionary Brotli only. On the second search, the metrics panel shows all three modes, with the delta compression bar in green. Keep searching to see how ratios vary across different queries. Note how the compression ratio spikes on duplicate or refresh queries.

Open DevTools → Network tab to inspect the real CDT headers (Use-As-Dictionary, Content-Dictionary, Vary) on every response. The JSON API provides raw measurement data.

Measurement Methodology

Compression is measured exclusively on the raw web page data. In modern dynamic websites, a large amount of bandwidth is consumed by hidden blocks of personalized, user-specific session data (such as user profiles, encrypted tokens, and personalized inventory lists) that are injected directly into the HTML.

To ensure rigorous and scientifically accurate measurements, the compression logic in this demo executes on the clean page prior to adding the visual metrics overlay. This ensures that the tracked ratios strictly reflect realistic, internet-scale web delivery behavior and are not artificially skewed by the byte footprint of the demonstration interface itself.

Note on Browser Support & Simulation: You can open DevTools → Network tab to inspect the real CDT headers (Use-As-Dictionary, Vary) attached to every response. However, because the Dictionary TTL extension is an emerging capability and may not be natively enabled in all browsers, this demonstration employs a seamless server-side fallback. We use a temporary cookie (_dict_hash) to simulate the browser sending the Available-Dictionary header. This guarantees the server can demonstrate the exact protocol flow and calculate accurate delta compression metrics, even if your browser has not yet implemented the TTL specification.
